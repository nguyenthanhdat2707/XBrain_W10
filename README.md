# W10 - Progressive Delivery with Analysis

GitOps setup for API deployment với Argo Rollouts + AnalysisTemplate.

## Concept

Deploy API với **canary strategy** và **automated analysis**:
- Rollout: 10% → 50% → 100%
- AnalysisTemplate query Prometheus để check success rate ≥ 95%
- Auto rollback nếu analysis fail
- AlertManager gửi email khi có SLO violation

## Requirements

- Docker Desktop
- kubectl
- minikube
- git

## Quick Start

### 1. Setup Cluster
```bash
minikube start -p w10 --driver=docker
kubectl config use-context w10
```

### 2. Install ArgoCD
```bash
kubectl create ns argocd
kubectl apply --server-side -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
kubectl -n argocd rollout status deploy/argocd-server
```

### 3. Access ArgoCD UI
```bash
# Port forward
kubectl -n argocd port-forward svc/argocd-server 8080:443 &

# Get password
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath='{.data.password}' | base64 -d; echo
```

## Challenge: Payments Tenant

The `payments` tenant is onboarded as a separate namespace managed by GitOps.

### GitOps apps

- `payments`: tenant infrastructure from `tenants/payments/`
- `payments-app`: workload from `apps/payments/`

Both apps are generated by the root App-of-Apps through `argocd/apps/`.

### Tenant resources

- Namespace: `payments`
- RBAC: `payments-dev` is bound to a namespaced Role only
- Resource controls: `ResourceQuota` and `LimitRange`
- Network isolation: default-deny, same-namespace allow, and DNS-only egress allow
- Workload: `payments-api` using signed image `ghcr.io/nguyenthanhdat2707/w10-api:0.0.2@sha256:ba92522c5d62e9ddabdddd63b0e3e49af99ce30fd8e4eb42e63a61eb1762aab9`

The minikube profile also runs `network-policy-engine`, a GitOps-managed kube-router DaemonSet in firewall-only mode, so Kubernetes `NetworkPolicy` resources are enforced at runtime.

### Why the old guardrails apply

Gatekeeper and Sigstore Policy Controller run at the admission layer. Every new workload in the cluster goes through those admission webhooks before it is stored by the Kubernetes API server. The policy decision is based on resource kind, namespace, image, and object fields, not on which team owns the app. Because `payments` is a normal namespace in the same cluster, the existing Gatekeeper constraints and Sigstore `ClusterImagePolicy` still apply.

The `payments` namespace has:

```yaml
policy.sigstore.dev/include: "true"
```

That label opts the namespace into Sigstore signature admission. This is safe here because the payments workload uses the already scanned and signed image `ghcr.io/nguyenthanhdat2707/w10-api:0.0.2`.

### Why Role/RoleBinding instead of ClusterRoleBinding

`Role` and `RoleBinding` keep tenant access scoped to one namespace. `payments-dev` can create and manage allowed workload resources in `payments`, but that binding does not grant permissions in `demo` or other namespaces. `ClusterRoleBinding` grants permissions at cluster scope, which makes it much easier for one tenant user to affect other teams.

### Payments checks

```bash
# RBAC
kubectl auth can-i create deployments.apps --as=payments-dev -n payments
kubectl auth can-i create deployments.apps --as=payments-dev -n demo
kubectl auth can-i get secrets --as=payments-dev -n payments
kubectl auth can-i update rolebindings.rbac.authorization.k8s.io --as=payments-dev -n payments

# Resource controls
kubectl get resourcequota,limitrange -n payments
kubectl apply --dry-run=server -f tests/payments/invalid-quota-pod.yaml

# Admission guardrails
kubectl apply --dry-run=server -f tests/payments/invalid-latest.yaml
kubectl apply --dry-run=server -f tests/payments/invalid-root-user.yaml
kubectl apply --dry-run=server -f tests/payments/invalid-hostnetwork.yaml
kubectl apply --dry-run=server -f tests/payments/invalid-replicas.yaml
kubectl apply --dry-run=server -f tests/payments/invalid-unsigned-image.yaml

# App health
kubectl get deploy,svc,pods -n payments
kubectl get applications -n argocd
```

## Test Scenarios (GitOps)

### Test 1: Successful Deployment (Success Rate ≥ 90%)
```bash
# Edit rollout to deploy with no errors
nano app-api/rollout.yaml
# Set: ERROR_RATE: "0"

git add app-api/rollout.yaml
git commit -m "test: deploy with 0% error rate"
git push origin main

# Watch AnalysisRun succeed
kubectl get analysisrun -n demo -w
```

### Test 2: Failed Deployment (Success Rate < 90%)
```bash
# Edit rollout to deploy with 15% error rate
nano app-api/rollout.yaml
# Set: ERROR_RATE: "0.15"

git add app-api/rollout.yaml
git commit -m "test: deploy with 15% error rate (should fail)"
git push origin main

# Watch AnalysisRun fail and auto rollback
kubectl get analysisrun -n demo -w
kubectl get rollout api -n demo
```

### Test 3: Trigger SLO Alert Email
```bash
# Edit rollout to set 10% error rate (triggers alert, but passes canary)
nano app-api/rollout.yaml
# Set: ERROR_RATE: "0.10"

git add app-api/rollout.yaml
git commit -m "test: deploy with 10% error rate (90% success)"
git push origin main

# Canary passes (≥90%) but SLO alert fires (below 95%)
# Wait 2-3 minutes, then check email inbox
```


## Configuration Reference

### Sync Waves
ArgoCD applications deploy in order:
- Wave -1: `app-common` (namespace)
- Wave 0: `k8s-prometheus`, `k8s-rollout` (infrastructure)
- Wave 1: `app-analysis`, `app-alert` (configuration)
- Wave 2: `app-api` (application)
